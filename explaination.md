# WandAI: Comprehensive Technical Explanation

This document provides a detailed overview of the technical architecture, design patterns, and implemented features of the **WandAI Multi-Agent Orchestration System**.

## 1. Project Overview

WandAI is an intelligent agentic system designed to take high-level natural language requests (e.g., "Research the latest EV trends and plot a market share graph") and autonomically execute them. Unlike simple chatbots, WandAI uses a **Multi-Agent Architecture** where specialized AI agents collaborate to solve complex problems.

The core philosophy is **Orchestration over Linear Chaining**: Instead of A -> B -> C, the system plans a dynamic dependency graph, allows for parallel execution, and supports cyclic refinement.

## 2. System Architecture

The project is split into a **Python FastAPI Backend** and a **React/Vite Frontend**, communicating via REST APIs and WebSockets.

### 2.1 The Backend (Python)
The brain of the operation. It hosts the LangGraph workflow and the agents.
- **Framework**: FastAPI (Async).
- **Orchestration**: `LangChain` + `LangGraph`.
- **LLM Provider**: Configurable (OpenAI/Gemini).
- **State Management**: In-memory `AgentState`.

### 2.2 The Frontend (React)
A "Mission Control" interface designed for visibility.
- **State Sync**: Uses WebSockets to stream the `AgentState` in real-time.
- **Artifact Rendering**: Renders Markdown, Code, and Charts generated by agents.
- **Plan Visualization**: Shows the execution plan as steps.

## 3. The Core: LangGraph Orchestration

At the heart of the system is `core/graph.py`, which defines the **State Machine** of the application.

### 3.1 The "Blackboard" State Pattern
We utilize a shared `AgentState` object (defined in `core/state.py`) that acts as a central blackboard.
- **Single Source of Truth**: All agents read inputs from and write outputs to this state.
- **Persistence**: LangGraph checkpoints this state at every step.
- **Components**:
  - `plan`: List of `PlanStep` objects (the to-do list).
  - `artifacts`: Dictionary of generated files/charts.
  - `logs`: Append-only execution log.
  - `messages`: Chat history for the LLM.

### 3.2 The Workflow Graph
The graph operates in a cyclic manner:
1.  **Orchestrator**: Analyzes `user_request` -> Generates `Plan`.
2.  **Dispatcher**: Looks at the `Plan` -> Decides which steps are ready to run (dependencies met).
3.  **Agents**: (Parallel Execution) Specialized agents run their assigned steps.
4.  **Reducer**: Aggregates results back into `AgentState`.
5.  **Loop**: Goes back to Dispatcher until all steps are `COMPLETED`.

## 4. The Agents

### 4.1. The Orchestrator (Manager)
- **Role**: Planner. Does not execute tasks.
- **Logic**: Uses a specialized LLM prompt to break requests into logical steps.
- **Output**: A DAG (Directed Acyclic Graph) of steps, e.g., `[Step 1: Research] -> [Step 2: Analyze Step 1]`.

### 4.2. Specialized Workers
Each agent has a specific "persona" and toolset:
- **Researcher**: Uses `Tavily API` to search the web. Returns markdown summaries.
- **Coder**: Uses a custom **Sandboxed Python Executor** (`exec()` with restricted globals) to run code, debug errors, and retry.
- **Analyst**: Generates charts using `matplotlib` (via code execution) and statistical insights.
- **Writer**: Compiles artifacts into a final report.

### 4.3. Custom Agents (The Builder)
We implemented a **Dynamic Agent System** (`custom_agents.json`).
- Users can define new agents (e.g., "Financial Advisor") via the UI.
- The Orchestrator is dynamically prompt-engineered to be aware of these new agents during planning.

## 5. Key Workflows

### 5.1 Dynamic Planning & Execution
Unlike static chains, the plan is dynamic.
- If the **Researcher** fails, the **Orchestrator** can modify the plan to retry or try a different strategy.
- **Optimistic UI**: The Frontend allows users to *manually* edit the plan (Add/Delete steps) before or during execution.

### 5.2 The Refinement Loop (Human-in-the-Loop)
We treat the entire process as a conversation.
- After a result is produced, the user can say "Make it blue" or "Add India data".
- The system preserves the **History** (artifacts/logs) and re-runs the Orchestrator with the new constraint + context.

### 5.3 Ambiguity Handling (Lazy Clarification)
If a request is vague ("Analyze the data"), the system pauses.
1.  **Detection**: Orchestrator calculates a `clarity_score`.
2.  **Suspension**: If score < 8, it enters `WAITING_CLARIFICATION` state.
3.  **Interaction**: Frontend polls this status, shows a Modal to the user.
4.  **Resumption**: User answers are injected into state, and execution resumes.

## 6. Technical Nuances & Challenges

### 6.1 Sandboxing Code execution
Allowing an LLM to write code is dangerous. We implemented a `code_executor` tool that:
- Runs in a separate process/thread logic (simulated).
- Restricts dangerous imports (like `os`, `sys`, `subprocess`).
- Captures `stdout`/`stderr` to feed back to the LLM for self-correction.

### 6.2 Real-time Visuals
We verified the usage of `lucide-react` icons and glassmorphism CSS to create a premium feel.
- **Streaming**: Logs appear instantly as agents "think".
- **Graph Updates**: The node graph (ReactFlow) updates live as steps change status (Pending -> In Progress -> Completed).

### 6.3 Resilience
- **JSON Parsing**: We hardened the `Analyst` agent to use Regex extraction because LLMs often "chat" around the JSON output.
- **Persistence**: We implemented `MemorySaver` to allow the graph to "sleep" while waiting for user input without blocking a thread.

## 7. Conclusion

WandAI represents a shift from "Chat with AI" to "Work with AI". By structuring the interaction around a shared Plan and Artifacts, we enable complex work that requires stability, visibility, and collaboration.
